# Обработка ошибок

Единообразная обработка ошибок — важная часть надёжного и
поддерживаемого backend-приложения. В BAD архитектуре принят строгий
подход к работе с ошибками, основанный на стандартах команды LAF и
лучших практиках NestJS. Для удобства и централизации используется
специальный утилитный класс `ErrorConstructor`, который позволяет
декларативно описывать ошибки и автоматически генерировать для них
коды, сообщения и готовые исключения.

## Основные принципы

1. **Ошибки только выбрасываются (throw), никогда не возвращаются.**  
   Функции не должны возвращать коды ошибок или специальные объекты
   вида `{ error: ... }`. Вместо этого в случае исключительной
   ситуации выбрасывается исключение.

2. **Использование встроенных исключений NestJS.**  
   Для HTTP-запросов используйте `HttpException` и его встроенные
   производные: `BadRequestException`, `UnauthorizedException`,
   `NotFoundException` и т.д. Это гарантирует правильный формат ответа
   и совместимость с фильтрами исключений.

3. **Централизованное описание ошибок через `ErrorConstructor`.**  
   Все специфичные для приложения ошибки рекомендуется определять с
   помощью `ErrorConstructor`. Это обеспечивает:
   - единый формат ошибок;
   - автоматическую генерацию уникальных кодов;
   - готовые экземпляры `HttpException` для выбрасывания;
   - удобную группировку по категориям.

4. **Глобальный фильтр ошибок.**  
   В проекте уже настроен `SentryGlobalFilter`, который автоматически
   отправляет все необработанные исключения в Sentry. При
   необходимости можно добавить свои фильтры.

## Структура папки `errors/`

Ошибки могут быть организованы как на глобальном уровне
(`src/errors/`), так и внутри конкретной версии (`src/v1/errors/`).
Выбор зависит от того, используются ли ошибки только в одной версии
API или во всём приложении.

- **`constructor.ts`** содержит класс `ErrorConstructor`, который
  используется для создания набора ошибок.
- **Категории** (папки) группируют ошибки по функциональности (auth,
  guards, strategies, database и т.д.).
- **Файлы ошибок** имеют суффикс `.errors.ts` и экспортируют объект с
  ошибками, созданный через `new ErrorConstructor(...).execute()`.

## Класс `ErrorConstructor`

Класс `ErrorConstructor` находится в `errors/constructor.ts`. Он
принимает префикс (например, `'AUTH GUARD'`) и объект с описаниями
ошибок. Каждая ошибка может быть задана тремя способами:

1. **Простая строка** — будет использована как сообщение.
2. **Расширенный объект** с полями:
   - `message`: краткое сообщение (обязательно);
   - `description`: подробное описание (обязательно);
   - `status`: HTTP-статус (по умолчанию `500`).
3. **Функция**, принимающая `puck` и возвращающая строку или
   расширенный объект (для динамического формирования сообщения).

После вызова `.execute()` возвращается объект, где каждому ключу
соответствует запись с полями:

- `code` — уникальный код ошибки (формируется автоматически);
- `puck` — идентификатор вида `"PREFIX KEY"`;
- `message` — полное сообщение (включает код и описание);
- `description` — описание;
- `exception` — готовый экземпляр `HttpException`, который можно сразу
  выбросить.

### Пример из `auth.errors.ts` (для стратегий)

```ts
import { HttpStatus } from "@nestjs/common";
import { ErrorConstructor } from "../constructor";

export const AUTH_STRATEGIES_ERRORS = new ErrorConstructor(
  "AUTH STRATEGY EXCEPTION",
  {
    AUTH_NOT_FOUND: {
      message: "Authorization not found",
      description: "Авторизация пользователя не была найдена в БД",
      status: HttpStatus.INTERNAL_SERVER_ERROR,
    },
    PASSWORD_ERROR: {
      message: "Password not equals",
      description: "Пароль не совпадает с паролем из БД",
      status: HttpStatus.FORBIDDEN,
    },
  },
).execute();

export default AUTH_STRATEGIES_ERRORS;
```

В результате `AUTH_STRATEGIES_ERRORS` содержит:

```ts
{
  AUTH_NOT_FOUND: {
    code: '...',
    puck: 'AUTH STRATEGY EXCEPTION AUTH_NOT_FOUND',
    message: '...',
    description: '...',
    exception: HttpException
  },
  PASSWORD_ERROR: { ... }
}
```

## Использование в коде

В сервисе, гарде или стратегии достаточно импортировать нужный объект
ошибок и выбросить `exception`:

```ts
import { AUTH_STRATEGIES_ERRORS } from "../errors/strategies/auth.errors";

// ...
if (!auth) {
  throw AUTH_STRATEGIES_ERRORS.AUTH_NOT_FOUND.exception;
}
```

Преимущества такого подхода:

- Не нужно каждый раз создавать `new HttpException(...)` с одними и
  теми же параметрами.
- Код ошибки автоматически добавляется в сообщение, что упрощает
  отладку.
- Описание ошибки хранится в одном месте и может быть использовано для
  документации.

## Логирование при регистрации

В конструкторе ошибок используется `logger.execute`, который при
инициализации модуля (при первом импорте) выводит в консоль сообщения
о зарегистрированных ошибках. Это помогает видеть, какие ошибки
доступны, и контролировать их коды.

## Рекомендации по категоризации

- Группируйте ошибки по функциональным модулям: `auth/`, `guards/`,
  `strategies/`, `database/`, `validation/` и т.д.
- Внутри каждой категории создавайте отдельный файл `.errors.ts`
  (например, `auth.errors.ts`), экспортирующий объект с ошибками.
- Если ошибки используются только в одном файле, можно определить их
  прямо там, но для переиспользования лучше выносить в общую папку.

## Интеграция с Sentry

Sentry уже подключена и настроена в проекте. Все необработанные
исключения автоматически попадают в Sentry благодаря глобальному
фильтру. Вы также можете вручную отправлять сообщения в Sentry,
используя `logger` из `@sentry/nestjs`:

```ts
import { logger } from "@sentry/nestjs";

logger.error("Something went wrong", { userId });
```

## Заключение

Правильная обработка ошибок делает приложение предсказуемым и
облегчает отладку. Придерживайтесь описанных правил:

- всегда выбрасывайте исключения;
- используйте встроенные HTTP-исключения NestJS;
- для специфичных ошибок применяйте `ErrorConstructor`, чтобы
  централизованно описать их с кодами и описаниями;
- группируйте ошибки по категориям в папке `errors/`;
- не забывайте про Sentry для мониторинга.
