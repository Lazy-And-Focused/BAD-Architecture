# Правила стиля и соглашения (Conventions)

Данный документ описывает основные соглашения по написанию кода в
проектах, использующих BAD архитектуру. Он базируется на общих
стандартах команды LAF и дополняет их правилами, специфичными для
нашего шаблона.

> **Полный список стандартов команды LAF доступен по ссылке:**
> [Стандарты написания кода “Лэйзи Энд Фокусед”](https://docs.laf-team.ru/agreements/general)

## 1. Общие принципы (отсылка к стандартам LAF)

Все базовые правила именования, форматирования, организации функций,
классов, импортов и работы с ошибками **строго соответствуют
стандартам LAF**. Пожалуйста, ознакомьтесь с ними по ссылке выше.
Ключевые моменты оттуда:

- **Наименование:** английский язык, нотации (`camelCase`,
  `UpperCamelCase`, `kebab-case`, `SCREAMING_SNAKE_CASE`), осмысленные
  имена, допустимые сокращения.
- **Функции:** единственная ответственность, до 3 аргументов, не более
  100 строк, предсказуемое имя (глагол или вопрос для boolean).
- **Классы:** `UpperCamelCase`, поля по умолчанию `private`, единый
  стиль исполнения (`execute`), методы без лямбда-синтаксиса.
- **Импорты:** сортировка (инициализаторы, типы, данные, файлы),
  `import type`, избегать `import *`.
- **Код:** пояснительные переменные, отсутствие магических чисел,
  фигурные скобки желательны.

## 2. Специфические правила BAD архитектуры

Помимо общих стандартов LAF, в проектах BAD необходимо соблюдать
следующие соглашения.

### 2.1. Структура файлов и папок

- **Имена файлов:** `kebab-case` с обязательным суффиксом, указывающим
  на тип: `.service.ts`, `.controller.ts`, `.module.ts`, `.guard.ts`,
  `.decorator.ts`, `.middleware.ts`, `.types.ts`, `.dto.ts`,
  `.utils.ts`, `.constants.ts`, `.errors.ts`, `.strategy.ts`.
- **Индексные файлы:** В каждой папке, предполагающей множественный
  экспорт, создавайте `index.ts` для реэкспорта публичных элементов.
  Это упрощает импорты (например, `@1/services`).

### 2.2. Переменные окружения

- Все переменные окружения описываются в файле `.env.example` с
  комментариями.
- Доступ к переменным осуществляется **только** через сервис
  `env.service.ts` (алиас `f@/env`), который обеспечивает типизацию и
  валидацию. Прямое использование `process.env` в коде запрещено.

### 2.3. Работа с ошибками

- Ошибки всегда выбрасываются (`throw`), а не возвращаются.
- В сервисах и контроллерах используйте встроенные исключения NestJS
  (`HttpException`, `BadRequestException` и т.д.).
- Для специфичных ошибок создавайте кастомные классы в папке `errors/`
  (глобальной или версионной). Имя файла должно иметь суффикс
  `.errors.ts`.

### 2.4. Контроль версий

- Lock-файлы (`pnpm-lock.yaml`, `package-lock.json`) **обязательно**
  хранятся в репозитории для обеспечения воспроизводимости сборок.
- Перед коммитом рекомендуется выполнять форматирование
  (`pnpm format`) и линтинг (`pnpm lint`).

### 2.5. Тестирование

- Модульные тесты размещаются рядом с тестируемым файлом и имеют
  суффикс `.test.ts` (например, `user.service.test.ts`).
- E2E-тесты находятся в папке `test/` в корне проекта.

## Заключение

Соблюдение данных соглашений обеспечивает единообразие кодовой базы,
упрощает вхождение новых разработчиков и уменьшает количество ошибок.
При возникновении вопросов обращайтесь к полной версии
[стандартов LAF](https://docs.laf-team.ru/agreements/general).
