# Структура проекта

BAD архитектура имеет строгую, но логичную организацию файлов. Понимание этой структуры необходимо для эффективной работы. Все основные исходные коды находятся в директории `src/`. Проект построен вокруг версионности API — каждая версия (v1, v2 и т.д.) изолирована в своей папке, что позволяет удобно развивать API, не ломая обратную совместимость.

## Директория `src/` — исходный код

Главная папка, где лежит весь код приложения. Внутри неё находятся как общие компоненты (доступные для всех версий), так и папки с конкретными версиями API.

```

├── app.module.ts          # Главный модуль приложения
├── main.ts                # Точка входа
├── {version}/             # Папка конкретной версии API (например, v1, v2)
│   ├── {version}.module.ts
│   ├── types/             # Типы, специфичные для версии
│   ├── services/          # Сервисы уровня версии
│   ├── middleware/        # Middleware, применяемые только в этой версии
│   ├── errors/            # Кастомные ошибки для версии
```

## Пояснения по ключевым элементам

### Версионирование (`{version}/`)

BAD изначально спроектирована для поддержки нескольких версий API. Папка с именем версии (например, `v1/`) содержит всё, что относится именно к этой версии: модуль, роуты, сервисы, типы и т.д. Это позволяет со временем добавлять `v2/` с новыми эндпоинтами, не трогая старую версию. Версионный модуль (например, `v1.module.ts`) регистрируется в главном `app.module.ts` с помощью `RouterModule`, чтобы все роуты версии имели префикс `/api/{version}`.

### Роуты (`routes/`)

Это самая важная часть — здесь живут эндпоинты вашего API. Каждый роут (например, `users`, `auth`) представляет собой отдельную папку. **Папки роутов не создаются вручную** — для этого используется генератор:

```bash
nest g bad auth # генератор для единственного числа: bad
nest g badp user # генератор для множественного числа: bad plural
```

Команда создаст типовую структуру роута со всеми необходимыми файлами. Внутри папки роута обязательно наличие:

- `dto/` — объекты передачи данных (Create, Update и т.д.).
- `{routeName}.controller.ts` — контроллер с обработчиками запросов.
- `{routeName}.controller.test.ts` — тест контроллера.
- `{routeName}.service.ts` — бизнес-логика.
- `{routeName}.service.test.ts` — тест сервиса.
- `{routeName}.module.ts` — модуль роута.
- `{routeName}.routes.ts` — файл, где задаются пути для контроллера (паттерн `ROUTE` и `ROUTES`).

Опционально могут быть тесты (`.test.ts`) и вложенные подроуты (например, `users/:userId/posts` — создаётся как отдельная папка внутри родительского роута). Максимальная вложенность роутов — 3–4 уровня.

### Глобальные и версионные компоненты

Многие папки (например, `middleware/`, `errors/`, `decorators/`) могут находиться как на глобальном уровне `src/`, так и внутри конкретной версии. Это сделано для гибкости:

- Если компонент (например, `AuthGuard`) нужен во всех версиях — кладите его в глобальную папку `guards/`.
- Если он специфичен только для версии `v1` — размещайте внутри `v1/guards/`.

Помеченные знаком `(?)` папки являются **необязательными** — если в них нет необходимости, их можно не создавать или удалить.

### Особые поддиректории

- `errors/` содержит кастомные классы ошибок. Обычно есть файл `constructor.ts` для базовой ошибки и папки по категориям (например, `auth/`, `database/`), где лежат конкретные ошибки с суффиксом `.errors.ts`.
- `utils/` — вспомогательные функции, общие для проекта или версии. В частности, есть готовые утилиты `create-endpoints.utils.ts` и `resolve-pathname.utils.ts`.

### Приложение и стратегии (`app/`)

Папка `app/` предназначена для компонентов, которые работают на уровне всего приложения, но не являются частью API как такового. Здесь живут:

- `session.app.ts` — настройка сессий (например, подключение express-session).
- `{appService}.app.ts` — прочие сервисы уровня приложения.
- `strategies/` — стратегии Passport.js для аутентификации (например, `google.strategy.ts`, `jwt.strategy.ts`).

## Файлы конфигурации (корень)

- `.dockerignore`, `dockerfile`, `docker-compose.yaml` — для контейнеризации.
- `.env`, `.env.{envName}` — переменные окружения (последние — для разных сред).
- `.gitignore`, `.prettierrc.mjs`, `eslint.config.mjs` — настройки инструментов.
- `nest-cli.json`, `tsconfig*.json` — конфигурация NestJS и TypeScript. **Менять осторожно**.
- `test/` — e2e-тесты.

## Итог

BAD архитектура предлагает чёткое разделение ответственности:

- **Версии** — для обратной совместимости.
- **Роуты** — для группировки эндпоинтов.
- **Глобальные компоненты** — для переиспользуемой логики.
- **Утилиты, типы, константы** — для поддержки кода.

Соблюдение этой структуры гарантирует, что проект останется понятным и поддерживаемым даже при активном росте и смене команд.
