# Конфигурация и окружение

Управление конфигурацией и переменными окружения — критически важная
часть любого backend-приложения. В BAD архитектуре принят
централизованный, строго типизированный подход к работе с окружением.
Это гарантирует, что все необходимые переменные заданы, проходят
валидацию и доступны в коде единообразно.

## Файлы окружения

В корне проекта находятся следующие файлы, связанные с окружением:

- **`.env`** — основной файл с переменными окружения для текущей среды
  (локальной, разработки и т.д.). **Никогда не должен попадать в
  репозиторий** (добавлен в `.gitignore`).
- **`.env.example`** — шаблон, содержащий все необходимые переменные с
  комментариями. Используется для быстрого создания `.env`.
- **`.env.{envName}`** (опционально) — файлы для конкретных сред,
  например `.env.production`, `.env.test`. Имя файла определяется
  переменной `NODE_ENV` или явно задаётся в `programm-mode`.

Загрузка переменных происходит с помощью библиотеки `dotenv`. Путь к
файлу определяется в `src/services/programm-mode/index.ts` через
`ENV_FILE_NAME` (например, при `NODE_ENV=development` загружается
`.env.development`, иначе — `.env`).

## Сервис конфигурации (`env.service.ts`)

Вся работа с переменными окружения централизована в сервисе,
расположенном в `src/services/env/` (экспортируется через алиас
`@/services`). Основные файлы:

- `config.constants.ts` — списки обязательных, уникальных и
  опциональных переменных, значения по умолчанию.
- `config.types.ts` — TypeScript-типы для конфигурации.
- `config.validators.ts` — валидаторы для уникальных переменных
  (например, проверка формата `TOKEN_EXPIRATION`).
- `config.transformers.ts` — функции преобразования и валидации
  переменных.
- `config.service.ts` — основной файл, который экспортирует объект
  `env`.

При импорте `@/services` происходит следующий процесс:

1. Загружаются переменные из соответствующего `.env`-файла.
2. Выполняется преобразование и валидация всех переменных согласно
   категориям.
3. Если какая-то обязательная переменная отсутствует или не проходит
   валидацию, приложение завершает работу с ошибкой (вызов
   `process.exit()`), а в консоль выводится подробное сообщение.
4. Формируется итоговый объект `env`, объединяющий:
   - значения по умолчанию (`DEFAULT_VALUES`),
   - переменные из `process.env`,
   - преобразованные и проверенные обязательные и уникальные
     переменные.

## Типизация переменных

Переменные окружения разделены на несколько категорий, что отражено в
типах:

- **Обязательные (`REQUIRED_VALUES`)** — переменные, которые должны
  быть обязательно заданы. Например, `DATABASE_URL`, `SESSION_SECRET`.
  Если они отсутствуют, приложение не запустится.
- **Уникальные (`UNIQUE_VALUES`)** — переменные, требующие
  дополнительной валидации (например, `TOKEN_EXPIRATION` должен быть в
  формате, понятном для `ms`). Для них определены специальные
  валидаторы.
- **Опциональные** — остальные переменные, для которых заданы значения
  по умолчанию в `DEFAULT_VALUES` (например, `PORT`, `ENCODING_TYPE`).
  Их можно не указывать в `.env`.
- **Переменные аутентификации** — динамически формируются для каждого
  типа `AuthTypes` (Google, GitHub и т.д.) с суффиксами `_CLIENT_ID`,
  `_CLIENT_SECRET`, `_CALLBACK_URL`. Например, `GOOGLE_CLIENT_ID`,
  `GITHUB_CLIENT_SECRET`.

Итоговый тип `Env` объединяет все категории и гарантирует, что в коде
будут доступны только корректно определённые поля.

## Валидация и трансформация

Для уникальных переменных применяются валидаторы из
`config.validators.ts`. Например, `TOKEN_EXPIRATION` проверяется
функцией `validateString` из `@/services`, которая убеждается, что
строка может быть преобразована в миллисекунды (поддерживаются форматы
типа `'28d'`, `'1h'`). Если валидация не пройдена, приложение
завершается с ошибкой.

Пример валидатора:

```ts
import { validateString } from "@/services/unit-time.service";

export const VALIDATORS: Validators = {
  TOKEN_EXPIRATION: validateString,
};
```

## Переменные для аутентификации

Для поддержки множества стратегий аутентификации (Passport.js) в
конфигурации предусмотрена генерация переменных на основе перечисления
`AuthTypes`. В `config.constants.ts` есть массив
`AUTH_DATA = ["CLIENT_ID", "CLIENT_SECRET", "CALLBACK_URL"]`. Для
каждого типа аутентификации (например, `google`, `github`) ожидаются
переменные вида:

- `GOOGLE_CLIENT_ID`
- `GOOGLE_CLIENT_SECRET`
- `GOOGLE_CALLBACK_URL`

Если какая-то из этих переменных отсутствует для используемой
стратегии, приложение также завершится с ошибкой.

Для удобного доступа к ним предусмотрена функция
`getPassportEnv(type: Uppercase<AuthTypes>)`, которая возвращает
объект с полями `id`, `secret`, `callback`. Это избавляет от ручного
формирования имён переменных.

```ts
import { getPassportEnv } from "@/services";

const googleConfig = getPassportEnv("GOOGLE");
// googleConfig = { id: '...', secret: '...', callback: '...' }
```

## Использование в коде

В любом файле проекта переменные окружения доступны через импорт
`@/services` (алиас настроен в `tsconfig.json`). Обычно импортируется
объект `env` или функция `getPassportEnv`.

```ts
import env, { getPassportEnv } from "@/services";

const port = env.PORT; // string, но со значением по умолчанию
const tokenExp = env.TOKEN_EXPIRATION; // уже проверено валидатором
```

Использование централизованного объекта гарантирует, что все
переменные загружены и типизированы. Не рекомендуется обращаться к
`process.env` напрямую.

## Добавление новой переменной окружения

Чтобы добавить новую переменную, выполните следующие шаги:

1. **Добавьте переменную в `.env.example`** с комментарием.
2. **Определите её категорию**:
   - Если переменная обязательна для работы приложения, добавьте её в
     массив `REQUIRED_VALUES` в `config.constants.ts`.
   - Если переменная требует специальной валидации, добавьте её в
     `UNIQUE_VALUES` и реализуйте валидатор в `config.validators.ts`.
   - Если переменная опциональна и имеет значение по умолчанию,
     добавьте запись в `DEFAULT_VALUES`.
   - Если переменная относится к аутентификации, она попадёт в
     динамическую генерацию через `AUTH_DATA` — обычно этого не
     требуется, если вы добавляете новый провайдер OAuth, просто
     добавьте тип в `AuthTypes`.
3. **Обновите тип**:
   - В `config.types.ts` дополните соответствующие типы
     (`RequiredProperties`, `UniqueProperties`, `PartialProperties`,
     `Env`), если необходимо.
4. **Используйте переменную** в коде через импорт `@/services`.

После изменений перезапустите приложение. При неверной конфигурации вы
увидите понятное сообщение об ошибке.
